## 一、总体架构

### 1.1 架构模式

```
┌──────────────────────────────────────────────────────┐
│                   Autoload 层                         │
│  GameManager (Node)     EventEngine (Node)            │
│  音频总管 (Node)                                       │
│  ↑ 持有 WorldState (Resource)                          │
├──────────────────────────────────────────────────────┤
│                   数据模型层 (全部 Resource)             │
│  WorldState ─→ CountryData / FactionData /             │
│                PoliticianData / EconomyData /           │
│                EmpireData / ModifierSlot /               │
│                TechState / DecisionState / WarData       │
│  EventDef ──→ EventOption / ExprNode / EffectNode       │
├──────────────────────────────────────────────────────┤
│                   场景 UI 层                            │
│  外交(3D地球) ← 经济/派系/事件/科研/保存/加载 (Control)   │
│  各场景通过 GameManager 信号刷新                         │
└──────────────────────────────────────────────────────┘
```

采用 **集中式单例 + 信号驱动 + Resource 序列化** 模式。这是 Godot 策略游戏的标准架构选型，合理且可维护。

### 1.2 与原版对比

| 维度 | 原版 Unity | 当前 Godot |
|------|-----------|-----------|
| 全局状态 | `GlobalScript.inst.gameState`（单例） | `GameManager.world`（Autoload + Resource） |
| 核心数据 | `int[] data = new int[150]` | `数值表: Array[int]`（200 槽位） |
| 序列化 | BinaryFormatter | Godot Resource (.tres) |
| 事件系统 | 180+ 独立 MonoBehaviour + KGEvent 双系统 | 统一 EventDef/ExprNode/EffectNode 系统 |
| UI 通信 | 直接访问单例 | 信号 + 直接访问混合 |
| 月度模拟 | `QueryChina.MonthlyTick()` + `TimeScript.Repaint()` | **缺失** |

## 二、做得好的部分

### 2.1 数据模型设计 — ★★★★★

`WorldState` 继承 `Resource`，核心是 `数值表: Array[int]`（200 个整数槽位），通过 `数值索引` 字典做语义映射。完美复刻了原版 `data[150]` 的设计，同时提供了 `get_data_value("party_support")` 这样的可读访问接口。

```gdscript
# world_state.gd — 数值表语义映射
const 数值索引 := {
    "party_support": 1, "党内支持": 1,
    "people_support": 3, "民众支持": 3,
    "money": 8, "budget": 8, "预算": 8,
    "industry": 12, "工业": 12,
    # ... 42 个别名覆盖全部常用索引
}
```

所有子对象（CountryData、FactionData、PoliticianData 等）也继承 Resource，使得存档/读档只需 `ResourceSaver.save()` / `ResourceLoader.load()` 一行代码即可完成——比原版的 BinaryFormatter 更干净。

### 2.2 事件系统 — ★★★★★

这是整个项目最成熟的子系统，质量接近 P 社游戏的 trigger/effect 系统：

**ExprNode（条件表达式树）**：组合模式，17 种原子条件类型 + AND/OR/NOT 嵌套组合。

```
ExprNode.Type 枚举:
  RESOURCE_AT_LEAST / RESOURCE_AT_MOST        → data[key] >= / <= value
  MODIFIER_ACTIVE / MODIFIER_INACTIVE         → modifies[key].active
  PREV_EVENT_DONE / PREV_EVENT_NOT_DONE       → event_done[id]
  PREV_EVENT_RESULT_IS                         → resultOfEvents[id] == value
  IS_FACTION_LEADER                            → data[56] == value
  EMPIRE_RELATION_AT_LEAST / AT_MOST          → empires[key].relations
  COUNTRY_HAS_TAG / COUNTRY_IS_SUBJECT_OF     → 国家标签/附庸检查
  DATE_AFTER / DATE_BEFORE                     → 日期范围
  HAS_FLAG / NOT_HAS_FLAG                      → global_flags[key]
  COUNTRY_EXISTS / RESOURCE_AT_LEAST_FOR       → 跨国检查
  ALL / ANY / NOT                              → 逻辑组合
```

**EffectNode（效果节点）**：18 种效果类型 + CUSTOM_SCRIPT 逃生舱口。

```
EffectNode.Type 枚举:
  ADD_RESOURCE / SET_RESOURCE                  → 数值表修改
  JOIN_ALLIANCE / LEAVE_ALLIANCE               → 联盟操作
  ADD_EMPIRE_RELATION / SET_EMPIRE_RELATION    → 超级大国关系
  ADD_FACTION_SUPPORT                           → 派系支持
  SET_WAR_STATE / SET_MODIFIER_ACTIVE          → 战争/修正
  SET_FLAG / CLEAR_FLAG / TRIGGER_EVENT        → 标记/事件链
  CUSTOM_SCRIPT                                 → GDScript 逃生舱口
```

**EventEngine 触发管线**：
```
每 tick → check_and_fire()
  ├── 事件链队列 → 立即触发队首
  ├── 待处理事件超时检查 → 强制触发
  └── 全事件扫描:
       ├── 即时事件 (mtth_base=0): 条件满足 → 立即触发
       └── 延时事件 (mtth_base>0): MTTH累积 → 待处理通知(10天缓冲) → 超时强制
```

这套系统的表达力足以描述原版全部约 300 个事件。已迁移 27 个事件定义（.tres 文件）。

### 2.3 数据加载层 — ★★★★

`DataLoader` 对原版 `GameStartScript.cs` 做了精确移植：

- 注释中标注了原版代码行号
- 19 个国家字段全部对应
- gwcode 修正机制（别名表 + contains 双向兜底）解决了原版内部索引 vs 国际标准编码的映射问题
- 后处理覆盖（`_apply_post_load_overrides`）复刻了原版的难度调整和初始化硬编码

### 2.4 通信解耦 — ★★★★

- `GameManager` ←信号→ `EventEngine`（`event_triggered` 信号，非直接调用）
- UI 场景通过 `world_state_loaded` / `date_changed` 信号刷新
- `WorldState` 的脏标记 `_economy_dirty` + `flush_economy()` 避免了每次修改都触发同步

### 2.5 地图渲染系统 — ★★★★★

- 16K×8K 区域 ID 编码贴图 + 双级调色板 GPU 查找
- 领土转移只需修改调色板中一个像素（O(1)），无需重绘底图
- 4 种地图模式（政体/影响力/军事联盟/经济联盟）
- 昼夜循环 + 23.4 度地轴倾斜模拟
- 射线拾取：屏幕坐标 → 3D 射线 → 球面交点 → UV → 像素 → 区域 ID → gwcode

## 三、核心缺陷

### 3.1 ★★★ 最严重：月度模拟循环完全缺失

原版 `QueryChina.MonthlyTick()` + `TimeScript.Repaint()` 是整个游戏经济/社会模拟的心脏。当前 Godot 版的 `tick()` 只做了：

```gdscript
# game_manager.gd:120 — 当前的 tick 实现
func tick():
    world.date.advance()           # 1. 推进日期
    EventEngine.check_and_fire()   # 2. 检查事件
    world.techs.advance_tick()     # 3. 科技研究
    world.flush_economy()          # 4. 同步经济显示
```

**以下全部缺失（共 17 个子系统）：**

#### 3.1.1 月度收入计算

原版每月自动增加预算收入：

```
月度收入 = 工业/15 + 农业/35 + 服务业/50 + Σ(贸易伙伴发展度/100)
data[8] += 月度收入
```

**当前状态**：预算永远不会自然增长，只有事件和手动操作能改变 data[8]。

#### 3.1.2 月度支出扣除

原版每月自动扣除：

```
data[8] -= data[69] / 40        # 债务利息
data[8] -= data[26] / 10        # 腐败税
if year >= 1980: data[8] -= 1   # 历史开支
if year >= 1983: data[8] -= 2   # 额外历史开支
```

**当前状态**：UI 中显示了债务损耗和腐败损耗的数值（经济.gd:175-188），但仅作为文字展示，不实际扣除。

#### 3.1.3 工业增长

原版每 3 个月根据工业支出和经济体制计算增长：

```
效率系数 = {10: 0.3, 11: 0.4, 12: 0.6, 13: 0.7, 14: 0.85, 15: 1.0}
增长量 = data[71] / 100 × 效率系数[data[16]]
每 3 月: data[12] += 增长量
```

这体现了原版核心的**改革两难悖论**：计划经济效率低(0.3)但政府控制力强，市场经济效率高(1.0)但预算池被 planka2 惩罚缩减。

**当前状态**：工业投入完全无效果，data[12] 永远不变。

#### 3.1.4 农业增长

```
每 3 月: data[13] += data[73] / 150   # 农业支出 / 150
```

**当前状态**：农业投入无效果。

#### 3.1.5 服务业增长

```
每 6 月:
  if data[36] > 50: data[68] += data[36] / 100   # 储备促进服务业
  data[68] += data[79] / 80                        # 服务业支出
  if data[16] >= 14: data[68] += 1                 # 市场经济加成
```

**当前状态**：服务业永不变。

#### 3.1.6 经济体制对经济的月度影响

原版有两套公式（1980年前/后），根据经济体制和外汇储备水平，对腐败、生活水平、服务业、工业产生正面或负面效果：

```
1980年前，鸟笼经济(13):
  腐败 -= 储备/400
  if 储备 < 600: 生活/服务/工业 -= 3 - 储备/150
  else: 生活/服务/工业 += 1

1980年后，混合经济(14):
  腐败 -= 储备/400
  if 储备 < 1500: 生活/服务/工业 -= 7 - 储备/150
  else: +3

1980年后，自由市场(15):
  腐败 -= 储备/200
  生活/服务/工业 -= 13 - 储备/150   # 除非储备极高，否则始终负面
```

这意味着市场化程度越高，需要的外汇储备越多才能维持正增长。

**当前状态**：完全缺失。

#### 3.1.7 腐败动态

```
if data[36] > 100: data[26] += 1       # 高储备增腐败
if data[16] >= 14: data[26] += 1       # 市场经济增腐败
if data[72] > 50: data[26] -= 1        # 高行政支出减腐败
```

**当前状态**：腐败永不变。

#### 3.1.8 生活水平漂移

```
每 6 月:
  目标值 = (工业 + 农业 + 服务业) / 30 - 腐败 / 5
  if data[5] < 目标值: data[5] += 1
  if data[5] > 目标值: data[5] -= 1
```

**当前状态**：生活水平静态不变。

#### 3.1.9 民众支持波动

```
每 10 月:
  if 生活水平 < 400: data[3] -= 1      # 生活水平低→民众不满
  if 生活水平 > 600: data[3] += 1      # 生活水平高→民众满意
```

**当前状态**：民众支持静态不变。

#### 3.1.10 党内支持与赤字

```
if data[8] < 0: data[1] -= 1           # 赤字→党内不满
```

**当前状态**：赤字无任何惩罚。

#### 3.1.11 预算支出效果

原版 `TimeScript.cs` 函数 `InfluenceFromInvestments()`（第 8081 行起）对 11 项预算分配**每月**执行一次效果，且每项都有 3~8 个子效果（含"不足惩罚"阈值）。详见附录 B 预算分配表。

| 支出项 | 核心效果 | 不足惩罚阈值 |
|-------|---------|-------------|
| 军费 data[71] | 军力增长(年份越晚成本越高) + 工业/团结附带 | <80: 工业-、生活- |
| 国安部 data[72] | **特工 += data[72]/10** (强效果) | 同时大幅降低自由/民众/党支持 |
| 科研 data[73] | 科研 += data[73]/50 | 无 |
| 行政 data[74] | **腐败 -= data[74]/20** (反腐核心) | 无 |
| 高层福利 data[75] | 党支持 += (data[75]-61)/5 | 仅>61有效 |
| 宣传 data[76] | 团结/民众(>70时) | <50: 腐败+/民众-/团结- |
| 农业 data[77] | **农业 += data[77]/15** | <40: 农业衰退 |
| 工业 data[78] | **工业 += data[78]/10** | <70: 工业衰退 |
| 服务 data[79] | **服务 += data[79]/10** | <40: 服务衰退 |
| 福利 data[80] | **生活 += (data[80]-40)/5** (强效果) | 基线40 |
| 外交 data[81] | 美苏关系改善 + 影响力扩散 | <60: 声望-1; =0: 声望-2 |

**当前状态**：所有预算分配都不产生任何游戏效果。玩家投入资金但看不到任何回报。

#### 3.1.12 人口增长

```
每年:
  生育政策=0（一胎）: 人口 += 人口/100    # 1%增长
  生育政策=1（二胎）: 人口 += 人口*2/100  # 2%增长
  生育政策=2（无限制）: 人口 += 人口*3/100 # 3%增长
```

**当前状态**：人口永不变。

#### 3.1.13 军力维护

```
if 军费 < 军力/10: data[22] -= 1        # 军费不足→军力衰退
```

**当前状态**：军队永不衰退。

#### 3.1.14 兵源计算

```
每 6 月:
  全民皆兵(30): 兵源 = 人口/10
  积极建军(31): 兵源 = 人口/15
  国防建设(32): 兵源 = 人口/25
  职业军队(33): 兵源 = 人口/40
```

**当前状态**：缺失。

#### 3.1.15 派系支持漂移

```
每 6 月:
  与政治路线匹配的派系: support += 2
  不匹配的派系: support -= 1
  盟友派系: support += 1
  被禁止的派系: support = max(0, support - 3)
```

**当前状态**：派系支持永不自动变化。

#### 3.1.16 满意度漂移

```
每年:
  目标值 = (党支持 + 民众支持 + 生活水平) / 30
  data[106] 向目标值漂移 ±1
```

**当前状态**：data[106]（满意现秩序者）永不变。

#### 3.1.17 政治体制自动重算

原版每 tick 根据所有制度维度的加权和重新计算 data[14]（政治体制类型）：

```
reform_score = (经济体制-9) + (政党制度-5) + (舆论政策-15) + (宗教政策-23)
             + (领土制度 + 军事学说 - 48) / 2
             + 特殊加成（OGAS +1, 晚期计划 +2）

reform_score ≤ 6  → 威权体制(0)
reform_score ≤ 9  → 保守社会主义(1)  // 需 data[16] ≤ 11
reform_score ≤ 11 → 有国家特色的社会主义(2)
reform_score ≤ 15 → 邓小平主义(3)    // 需 data[16] > 11
reform_score ≤ 20 → 社会民主主义(4)
其余             → 自由主义(5)
```

**当前状态**：政体类型从加载后永不自动更新。

#### 3.1.18 赤字处理与储备吸收

原版中预算为负时，自动从储备金中补充：

```
if data[8] < 0:
  if data[36] 足够: data[36] += data[8]; data[8] = 0
  else: 耗尽储备，强制暂停并打开经济界面
```

**当前状态**：缺失，赤字只是数字显示。

#### 3.1.19 年度重置

原版每年 1 月 1 日：

```
所有 party_number[] /= 10     # 派系席位衰减
data[106] /= 10                # 满意者衰减
所有国家 soc_stab 重置为 0     # 社会稳定重置
政治家年龄 += 1                 # 可能触发自然死亡
```

**当前状态**：缺失。

### 3.2 ★★ UI 直接写 WorldState

`GameManager` 头部注释声明自己是"唯一的数据写入者"，但实际上：

```gdscript
# 经济.gd:87 — 直接写数值表
w.数值表[idx] += delta
w.数值表[BUDGET_IDX] -= delta

# 派系.gd:306-308 — 直接写数值表
w.数值表[8] -= diff * DEDUCT_BUDGET
w.数值表[5] -= diff * DEDUCT_LIVING
w.数值表[1] -= diff * DEDUCT_PARTY
```

这些操作绕过了 `GameManager`，导致：
- 无法统一拦截/验证数值变更
- 无法发出变更信号通知其他正在显示的 UI
- 未来加入存档校验或回放功能时会遇到困难

### 3.3 ★ flush_economy 调用不统一

经济场景调用 `w.sync_economy()`，派系场景调用 `w.flush_economy()`。两者最终都同步 EconomyData，但入口不一致容易造成混淆。

### 3.4 ★ 缺少数值变更通知

预算分配、政策切换等操作修改数值表后，没有信号通知其他场景。当前靠场景切换时重新 `_refresh()` 解决，但如果将来同屏显示多个面板（如小地图 + 状态栏 + 经济面板），会出现数据不同步。

## 四、数值设计评估

### 4.1 数值表内部精度

数值表以 ×10 精度存储整数，UI 显示时除以 10.0。例如内部 `数值表[1] = 500` 代表党内支持度 50.0%。这是原版的做法，移植正确。

```gdscript
# economy_data.gd:35 — 显示层缩放
func sync(数值表: Array[int]) -> void:
    党内支持度 = float(数值表[1]) / 10.0
    民众支持度 = float(数值表[3]) / 10.0
    # ...
```

### 4.2 planka（预算上限）计算

经济.gd:65-71 中的 planka 计算正确复刻了原版逻辑：

```gdscript
func _calc_planka(w: WorldState) -> int:
    var total: int = w.数值表[BUDGET_IDX] + w.数值表[RESERVE_IDX]
    for i in range(71, 82):
        total += w.数值表[i]
    if w.数值表[16] > 12:
        total -= (w.数值表[16] - 12) * (total / 10)
    return total
```

设计意图：经济体制从国家资本主义(12)向市场经济过渡时，政府对预算的控制力下降——每高一级，总可分配预算池减少 10%。单类别上限为 `planka / 6`。

### 4.3 政策切换成本

派系.gd:104-109 中的政策切换成本正确复刻了原版 `Doctrine_button_script.cs`：

```
需求检查: 预算+储备 >= diff*50, 党支持 >= diff*300
扣减: 预算 -= diff*50, 生活水平 -= diff*50, 党支持 -= diff*30
```

但原版还有以下效果当前缺失：
- 外交声望惩罚：`data[6] -= delta * 20~60`（按类别不同）
- 改革积分：`data[33] += delta * 100`（经济制度）、`data[55] += delta * 100`（政党制度）
- 自由化增加：`data[4] += delta * 10`（一党制）或 `delta * 20`（多党制）
- 改革不稳定：`data[92] += delta * 15`
- 前置条件：毛死亡（`data[38] >= 100`）、正确派系掌权（`data[56]`）

### 4.4 经济体制效率系数（缺失）

原版中工业增长效率与经济体制挂钩，这是核心的改革两难设计：

| data[16] 值 | 经济体制 | 工业效率 | 腐败倾向 | planka 惩罚 | 储备门槛(1980后) |
|---|---|---|---|---|---|
| 10 | 苏式计划经济 | 0.3 | 低 | 0% | — |
| 11 | 中式计划经济(OGAS) | 0.4 | 低 | 0% | — |
| 12 | 国家资本主义 | 0.6 | 中 | 0% | 600 |
| 13 | 国控资本主义(鸟笼) | 0.7 | 中-高 | -10% | 750 |
| 14 | 混合经济 | 0.85 | 高 | -20% | 1500 |
| 15 | 自由市场 | 1.0 | 很高 | -30% | ~2000 |

市场化带来高效率但也带来更高的腐败和储备需求。这个设计让玩家必须在效率和稳定之间做艰难抉择。

### 4.5 贷款机制

经济.gd 中的贷款机制部分正确：

```gdscript
# 贷款上限 = (美国关系 + 苏联关系) / 5
func _loan_limit(w: WorldState) -> int:
    return (usa_rel + ussr_rel) / 5

# 借入: 债务+N, 预算+N, 党支持-N
# 还款: 债务-N, 预算-N, 党支持-N
```

但原版还有以下机制缺失：
- 贷款前提条件：需有经济联盟(econ/SEV)或无 modifier[12]，且需 science[9] 或在 SEV/ASEAN 中
- 债务自动减少：`if data[69] > 10: data[69] -= data[69]/40/2 + 1`
- 美国从中国债务获利：`if data[69] > 7: empires[0].money += data[69]/7`
- 经互会内债务影响苏联关系：`empires[1].relations -= data[69]/20`

### 4.6 贸易伙伴计算（缺失）

原版每 tick 重新计算贸易伙伴：

```
data[23] = data[70]（基础贸易值）
for 每个同经济联盟国家 或 亲中/有贸易关系的国家:
    data[23] += 1~3（按其政体类型加成不同）
data[25] = 贸易伙伴数量
```

贸易收入直接影响月度预算收入。

## 六、通信处理评估

### 6.1 当前通信拓扑

```
GameManager ──signal──→ UI 场景 (date_changed, world_state_loaded, event_started)
EventEngine ──signal──→ GameManager (event_triggered)
EventEngine ──signal──→ 外交场景 (event_notification, event_notification_dismissed)
```

### 6.2 优点

- `EventEngine` → `GameManager` 之间用信号解耦，避免了双向 Autoload 依赖
- 场景切换用 UID（`uid://...`），不受文件重命名影响
- 地球选择信号链：`TerritoryMap.country_selected` → `外交.gd._on_country_selected` → `GameManager.select_country`

### 6.3 问题

1. **数据写入不统一**：UI 直接写 `数值表[]`，绕过 GameManager（详见 3.2）
2. **缺少变更通知**：数值表修改后无信号广播，多面板同屏时会数据不同步
3. **sync_economy / flush_economy 混用**：两个入口做同一件事，应统一

## 附录 A：严重 BUG — 预算项索引映射错误

### 问题描述

```gdscript
# ── 经济体制对经济的月度影响 ──
func _biannual_econ_system_effect(w: WorldState, year: int) -> void:
    var reserve := w.数值表[36]
    var econ := w.数值表[16]

    if econ == 12:  # 国家资本主义
        w.数值表[26] -= reserve / 200
        var delta := 3 - reserve / 150 if reserve < 600 else -1
        w.数值表[5] -= delta
        w.数值表[68] -= delta
        w.数值表[12] -= delta

    elif econ == 13:  # 鸟笼经济
        var threshold := 600 if year < 1980 else 750
        var base_loss := 3 if year < 1980 else 4
        w.数值表[26] -= reserve / (400 if year < 1980 else 600)
        if reserve < threshold:
            var delta := base_loss - reserve / 150
            w.数值表[5] -= delta
            w.数值表[68] -= delta
            w.数值表[12] -= delta
        else:
            w.数值表[5] += 1
            w.数值表[68] += 1
            w.数值表[12] += 1

    elif econ == 14:  # 混合经济
        w.数值表[26] -= reserve / 400
        if year >= 1980:
            if reserve < 1500:
                var delta := 7 - reserve / 150
                w.数值表[5] -= delta
                w.数值表[68] -= delta
                w.数值表[12] -= delta
            else:
                w.数值表[5] += 3
                w.数值表[68] += 3
                w.数值表[12] += 3

    elif econ == 15:  # 自由市场
        w.数值表[26] -= reserve / 200
        var delta := 13 - reserve / 150   # 除非储备极高否则始终负
        w.数值表[5] -= delta
        w.数值表[68] -= delta
        w.数值表[12] -= delta


# ── 政变检查 ──
func _coup_check(w: WorldState) -> void:
    # 公式: 党支持 <= 300 + 自由化/5 - (民众支持-500)/5
    var threshold := 300 + w.数值表[4] / 5 - (w.数值表[3] - 500) / 5
    if w.数值表[1] <= threshold:
        pass  # TODO: 触发政变事件


# ── 政治体制自动重算 ──
func _political_system_recalc(w: WorldState) -> void:
    var score: int = (w.数值表[16] - 9) + (w.数值表[15] - 5) \
                   + (w.数值表[17] - 15) + (w.数值表[50] - 23) \
                   + (w.数值表[18] + w.数值表[51] - 48) / 2
    if w.数值表[16] == 11: score += 1      # OGAS 加成
    if w.数值表[16] == 10: score += 2      # 晚期计划加成

    var new_system: int
    var new_gosstroy: int
    if score <= 6:
        new_system = 0; new_gosstroy = 0   # 威权体制
    elif score <= 9 and w.数值表[16] <= 11:
        new_system = 1; new_gosstroy = 1   # 保守社会主义
    elif score <= 11:
        new_system = 2; new_gosstroy = 1   # 国特社会主义
    elif score <= 15 and w.数值表[16] > 11:
        new_system = 3; new_gosstroy = 2   # 邓小平主义
    elif score <= 20:
        new_system = 4; new_gosstroy = 3   # 社会民主主义
    else:
        new_system = 5; new_gosstroy = 3   # 自由主义

    w.数值表[14] = new_system
    var pc := w.get_player_country()
    if pc: pc.government = new_gosstroy
```
